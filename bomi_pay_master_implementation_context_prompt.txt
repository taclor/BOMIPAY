4. MANDATORY ENGINEERING PRINCIPLES
==================================================

1. Financial correctness over speed
2. Security over convenience
3. Auditability over hidden magic
4. Idempotency everywhere money moves
5. Append-only event history for financial state
6. Explicit state transitions only
7. Deterministic workflows for money-impacting actions
8. Multi-tenant safety by default
9. Observability built in, not added later
10. Fail closed, not open
11. Least privilege everywhere
12. No silent data loss
13. No hidden mutation by AI agents
14. Every external provider integration must be isolated behind an adapter
15. Every important money-related action must be replayable or traceable

==================================================
5. NON-NEGOTIABLE NO-GO RULES
==================================================

Never do any of the following:

- Never hardcode API keys, secrets, tokens, or credentials
- Never store provider secrets unencrypted
- Never log raw secrets, tokens, full card-like data, or sensitive credentials
- Never allow AI to directly mutate ledger or payment state without controlled workflow authorization
- Never use floating point for money values
- Never treat financial events as overwrite-only records
- Never skip idempotency for write operations
- Never update transaction history destructively
- Never trust provider payloads without verification
- Never accept webhook calls without signature validation where supported
- Never build one-off provider logic directly into core services
- Never merge business logic into controllers when it belongs in domain services
- Never bypass audit logging for admin or finance operations
- Never use “best effort” persistence for transaction events
- Never silently swallow exceptions in payment flows
- Never build the system around one provider’s quirks
- Never return vague error messages for internal teams when structured errors are possible
- Never make architecture decisions that block future routing, ledgering, or dispute workflows
- Never let AI-generated code enter production without tests and review
- Never prioritize nice UI over operational correctness

==================================================

Follow this implementation direction:

Frontend:
- Next.js
- TypeScript
- clean operational UI
- dashboard-first, not marketing-first

- strong typing

- canonical internal transaction model mandatory
- append-only event tables mandatory

10. SECURITY RULES
==================================================

Implement these from the start:

- secure password hashing
- JWT or session security with refresh handling
- RBAC enforcement at route and service levels
- encrypted storage of provider credentials
- secrets from environment or secret manager only
- HMAC verification for inbound webhooks
- rate limiting on sensitive endpoints
- payload validation for every request
- MFA-ready auth design
- tenant isolation in all queries
- audit logging for all privileged actions
- suspicious activity monitoring hooks
- replay protection for webhook ingestion
- input/output validation with typed schemas
- masked logging for sensitive fields

==================================================
11. FINANCIAL INVARIANTS
==================================================

These rules must be preserved in code:

- Money values must be stored in minor units or exact decimals
- Every transaction state change must be traceable
- Event history must be append-only
- Reprocessing the same provider event must not duplicate effects
- Provider payload and normalized transaction state must both be preserved
- Ledger-affecting actions must be explicit and auditable
- Reconciliation must never mutate source truth invisibly
- Dispute workflows must preserve the complete timeline
- AI suggestions must never be treated as authoritative financial truth
- Human or workflow approval is required before any irreversible financial action

==================================================
12. AI USAGE RULES
==================================================

AI is an assistant layer, not an uncontrolled operator.

Allowed:
- explain transaction failures
- summarize reconciliation mismatches
- generate support drafts
- classify likely incident causes
- surface anomaly patterns
- recommend next actions

Not allowed without explicit workflow controls:
- marking funds as settled
- changing ledger balances
- reversing payments
- closing disputes automatically without policy support
- changing risk outcomes silently
- modifying provider credentials
- overriding transaction truth

All AI outputs must be:
- retrieval-grounded
- traceable to internal records
- confidence-labeled where appropriate
- safe to reject
- non-destructive by default
